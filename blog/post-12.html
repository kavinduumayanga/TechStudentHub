How Hosting Works (GitHub Pages Explained)

Hosting means putting your website files on a server so other people can access them. GitHub Pages is a beginner-friendly way to host static sites for free. This guide explains the key concepts and the #1 cause of broken links on Pages: using the wrong kind of URL paths.

What hosting is

When you open a website, your browser requests files (HTML/CSS/JS/images) from a server. Hosting is the service that stores those files and serves them to visitors.

A static site is simple: you upload files, and the server returns them when requested.

Domains and URL paths

A URL has a domain and a path. Example:

  • Domain: example.com
  • Path: /blog/post-1.html

When you use a link that starts with /, it usually means “from the domain root.” That’s where beginners get stuck on GitHub Pages project sites.

GitHub Pages basics

GitHub Pages publishes your site from a repository branch. After you push changes, GitHub builds and serves your site at a Pages URL.

For static sites, this is convenient: you can host HTML/CSS/JS without a server or database.

Project site vs user site (important)

GitHub Pages supports two common site types:

  • User/organization site: served from the domain root
  • Project site: served from a sub-path based on the repository name

If your site is a project site, your deployed URL contains the repo name (a subfolder). In that case, root-based links like /blog/index.html can break because they point to the domain root instead of your project folder.

Relative links (the safe approach)

Relative links do not start with /. They are resolved based on the current page’s folder.

Examples:

  • From the site homepage: blog/index.html
  • From a blog post inside the blog folder: index.html (same folder)
  • From a blog post back to home: ../index.html

This site uses relative links so it works reliably as a project site.

A checklist to avoid 404 errors

  • Avoid leading-slash internal links on project sites.
  • Check every navigation link on every starting layout.
  • Use consistent file names (don’t mix Index.html and index.html).
  • Verify assets exist (images and CSS paths).
  • Add a sitemap and update it when you add new pages.
  • Test 404 page so users aren’t stuck.
