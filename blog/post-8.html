How Git and GitHub Work (Simple Explanation)

Git and GitHub are often taught with lots of commands and confusing terms. You don’t need that to start. What you need is a mental model: Git is your project’s history, and GitHub is a place to store and share that history. This guide explains the parts that matter for beginners—especially if you’re building a static website.

The mental model (Git vs GitHub)

  • Git is a tool on your computer that tracks changes in a folder.
  • GitHub is a website that hosts Git repositories online.

That’s it. Everything else is built on top of those two ideas.

Commits: save points you can trust

A commit is like a “snapshot” of your project at a point in time. You can always go back to a commit if you break something.

Good commit habits

  • Commit small changes that you can describe in one sentence.
  • Write messages that explain intent (e.g., “Fix blog links for project site”).
  • Don’t commit secret keys or private files.

Branches: try changes safely

A branch is a separate line of history. Beginners can keep it simple:

  • Use main as your stable branch.
  • Create a branch when you want to try big changes.

If you’re working solo, you can still use branches to reduce fear: you’ll feel safer experimenting.

Remotes: your copy vs GitHub’s copy

Your project exists in two places:

  • Local repository (on your computer)
  • Remote repository (on GitHub)

The two key actions:

  • Push: send your commits to GitHub
  • Pull: bring GitHub commits down to your computer

A simple beginner workflow

For a static site, a beginner workflow looks like this:

  1. Edit files.
  2. Preview locally (open HTML files in a browser).
  3. Stage changes: git add .
  4. Commit: git commit -m "Describe the change"
  5. Push: git push

That’s enough to be productive and safe.

How this connects to GitHub Pages

GitHub Pages builds your site from your repository. When you push changes, Pages updates the deployed site shortly after.

If you’re learning hosting, read How hosting works (GitHub Pages explained). If you’re learning how to build the site itself, read How to build your first static website.

Common mistakes and how to avoid them

  • Forgetting to pull (when collaborating): pull before starting work.
  • Committing everything: don’t commit downloads, build artifacts, or large random files.
  • Huge commits: if a commit touches many unrelated files, it’s hard to debug later.
  • Not writing messages: “update” doesn’t help future-you.
